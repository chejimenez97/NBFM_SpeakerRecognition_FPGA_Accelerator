# FINN Speaker Recognition — FPGA-Accelerated CNN on NBFM Signals

FPGA-accelerated CNN speaker recognition on Narrow Band FM IQ signals using the FINN framework, Brevitas QAT, and Kria KV260.

## Target Hardware

| Parameter | Value |
|-----------|-------|
| FPGA Part | `xck26-sfvc784-2LV-c` (Kria KV260 / K26 SOM) |
| Clock | 200 MHz (5 ns period) |
| HLS Backend | Vitis HLS 2023.2 |
| Shell Flow | VIVADO_ZYNQ (Zynq UltraScale+) |

## Project Files

| File | Description |
|------|-------------|
| `MFCC_dataset_generation.ipynb` | Generates MFCC dataset splits from raw IQ files |
| `MFCC_CNN_model_training.ipynb` | TensorFlow reference CNN (architecture exploration) |
| `FINN_Speaker_Recognition_Synthesis.ipynb` | **Main notebook** — trains QAT models, exports ONNX, runs FINN synthesis |
| `setup_env.sh` | Installs FINN/Brevitas/QONNX deps inside the Docker container |
| `folding_config.json` | Auto-generated by the notebook; sets SIMD=4 for MVAU nodes |
| `sr_cnn_requirements.txt` | Extra pip dependencies (librosa, seaborn, tensorflow-cpu) |

## CNN Architecture

Compact 3-block CNN with square pooling, ported from the TF reference:

```
Input (2, 24, 288)  ← zero-padded to multiples of 8
  → Conv2d(8, 3×3)  → BN → QuantReLU → MaxPool(2×2)   → (12, 144)
  → Conv2d(16, 3×3) → BN → QuantReLU → MaxPool(2×2)   → (6, 72)
  → Conv2d(16, 3×3) → BN → QuantReLU → MaxPool(2×2)   → (3, 36)
  → Flatten (1728) → QuantLinear(32) → QuantReLU → QuantLinear(NUM_CLASSES)
```

**Model Variants:**

| Variant | Weights | Activations | FINN Synthesis |
|---------|---------|-------------|----------------|
| Float | FP32 | FP32 | No — reference only |
| QAT-8 | INT8 | UINT8 | Yes |
| QAT-4 | INT4 | UINT8 | Yes |

## Environment Setup — Complete Step-by-Step Guide

### Prerequisites

- Linux host (Ubuntu 20.04+ recommended)
- Docker CE installed
- Xilinx Vitis/Vivado 2023.2 installed at `/tools/Xilinx`
- At least 16 GB RAM (32 GB recommended for synthesis)
- FINN repository cloned (e.g., at `~/finn`)

### Step 1: Set Host Environment Variables

Add these to your `~/.bashrc`:

```bash
export FINN_XILINX_PATH="/tools/Xilinx"
export FINN_XILINX_VERSION="2023.2"
```

Then reload:

```bash
source ~/.bashrc
```

These tell FINN's `run-docker.sh` to mount the Xilinx tools into the container.

### Step 2: Verify FINN Installation

```bash
cd ~/finn
bash run-docker.sh quicktest
```

This runs a short test suite in a temporary container. Wait for it to complete — you should see `977 passed, 0 failed` or similar. This container is destroyed after the test.

### Step 3: Launch the FINN Docker Container

```bash
cd ~/finn
bash run-docker.sh bash
```

This creates a container named `finn_dev_<username>`. **Important:** This is a separate container from quicktest. Every `run-docker.sh` invocation creates an independent container with the `--rm` flag (destroyed on stop).

Leave this terminal open — the container is now running.

### Step 4: Fix Container Permissions (From a Second Host Terminal)

Open a **second terminal** on the host and run:

```bash
docker exec -u root finn_dev_$(whoami) bash -c \
  "chown -R $(id -u):$(id -g) /home/$(whoami) && chmod 755 /home/$(whoami)"
```

This fixes permission errors when VS Code attaches to the container or when pip installs packages.

**Do NOT use `FINN_DOCKER_RUN_AS_ROOT=1`** — it causes other issues.

### Step 5: Install FINN Dependencies (From Host Terminal)

FINN's Docker entrypoint often fails silently because `setuptools-scm` can't find `.git` metadata in `deps/`. You must manually install from the host terminal:

```bash
bash setup_env.sh
```

The `setup_env.sh` script runs the following inside the container as root:

```bash
#!/bin/bash
# setup_env.sh — Run from HOST terminal after every container start
CONTAINER=${1:-finn_dev_$(whoami)}

docker exec -u root $CONTAINER bash -c "
  cd /home/*/finn

  # Install all FINN dependencies (deps/ subdirectories)
  for dep in deps/*/; do
    if [ -f \"\$dep/setup.py\" ] || [ -f \"\$dep/pyproject.toml\" ]; then
      name=\$(basename \$dep)
      SETUPTOOLS_SCM_PRETEND_VERSION=0.0.0 pip install -e \"\$dep\" || echo \"WARNING: \$name failed\"
    fi
  done

  # Install FINN itself
  SETUPTOOLS_SCM_PRETEND_VERSION=0.10.1 pip install -e .

  # Install project-specific requirements
  cd SR_PROJECT && pip install -r sr_cnn_requirements.txt

  # Source Vitis HLS tools into the container's default bash profile
  if [ -f /tools/Xilinx/Vitis_HLS/2023.2/settings64.sh ]; then
    echo 'source /tools/Xilinx/Vitis_HLS/2023.2/settings64.sh' >> /etc/bash.bashrc
    echo 'Vitis HLS sourced into bash profile.'
  fi

  echo 'Done.'
"
```

**You must run this after every container start** — the `--rm` flag destroys everything on stop.

### Step 6: Verify Installations

From the host terminal:

```bash
# Check FINN, Brevitas, QONNX
docker exec finn_dev_$(whoami) pip show brevitas qonnx finn

# Check Vitis HLS
docker exec finn_dev_$(whoami) bash -c \
  "source /tools/Xilinx/Vitis_HLS/2023.2/settings64.sh && which vitis_hls"
```

You should see version info for all three packages, and a path like `/tools/Xilinx/Vitis_HLS/2023.2/bin/vitis_hls`.

### Step 7: Attach VS Code to the Container

1. Open VS Code on the host
2. Install the **Dev Containers** extension
3. `Ctrl+Shift+P` → **Dev Containers: Attach to Running Container**
4. Select `finn_dev_<username>`
5. Open the folder `/home/<username>/finn/SR_PROJECT`
6. Select the **Python 3** kernel (system default — do NOT create a venv)

### Step 8: Place Project Files

Copy all project files into the FINN container's project directory:

```
finn/
  SR_PROJECT/
    FINN_Speaker_Recognition_Synthesis.ipynb
    sr_cnn_requirements.txt
    setup_env.sh
    MFCC_datasets/
      metadata.json
      element_spec.pkl
      train_batches/
      validation_batches/
      test_batches/
```

The `MFCC_datasets/` folder must be generated first by running `MFCC_dataset_generation.ipynb`.

### Step 9: Clean Previous Artifacts (If Retraining)

If you're running the notebook after a previous run or architecture change:

```bash
rm -rf trained_models/* onnx_exports/* finn_build/* training_history/*
```

### Step 10: Run the Notebook

Open `FINN_Speaker_Recognition_Synthesis.ipynb` in VS Code and run all cells. The notebook will:

1. Load MFCC splits from `MFCC_datasets/`
2. Pad inputs to (24, 288) for FINN MaxPool divisibility
3. Train Float, QAT-8, and QAT-4 models
4. Export QONNX models
5. Source Vitis HLS tools into the Python environment (cell 7a)
6. Generate a folding config with SIMD=4 (cell 7b)
7. Run FINN synthesis for 4-bit then 8-bit models (~2–4 hours each)

## Requirements File

`sr_cnn_requirements.txt` should contain only packages not already in the FINN container:

```
librosa>=0.10.0
seaborn>=0.12.0
tensorflow-cpu
```

**Never include** in this file: `numpy`, `torch`, `brevitas`, `onnx`, `ipykernel`. These are managed by FINN's dependency tree. In particular, upgrading numpy breaks Brevitas due to C ABI incompatibility.

## Known Issues and Solutions

### Container loses all state after stop/restart

**Cause:** FINN's `run-docker.sh` uses `--rm`, which destroys the container on exit.
**Fix:** After every `bash run-docker.sh bash`, re-run `bash setup_env.sh` from the host terminal.

### `brevitas` / `qonnx` / `finn` not found after container start

**Cause:** FINN's entrypoint script fails silently because `setuptools-scm` can't find `.git` metadata.
**Fix:** Run `bash setup_env.sh` from a host terminal. Verify with `pip show brevitas qonnx finn`.

### Permission denied during pip install

**Cause:** Container user has no write access to system Python packages.
**Fix:** Use `docker exec -u root` from the host terminal (this is what `setup_env.sh` does). Never use `sudo` inside the container.

### `vitis_hls not found in PATH` (step 12 — step_hw_ipgen)

**Cause:** Vitis HLS settings not sourced in the Python kernel's environment. The notebook runs inside a Jupyter kernel that doesn't inherit the container's bash profile.
**Fix:** The notebook includes a cell (7a) that programmatically sources `/tools/Xilinx/Vitis_HLS/2023.2/settings64.sh` into `os.environ` using `subprocess`. If the path differs on your system, update the `vitis_settings` variable in that cell.

### `Unable to load Tcl app xilinx::xcelium` (step 12/13)

**Cause:** Vivado tries to load the Cadence Xcelium simulator during `export_design`, but Xcelium is not installed. This causes HLS IP generation to fail silently for some nodes, which then triggers "IP definition not found" errors at the stitching step.
**Diagnosis:** Check the HLS log for the failing node:

```bash
find /tmp/finn_dev_$(whoami)/ -name "*.log" -path "*StreamingMaxPool*"
```

**Fix (under investigation):** Check the generated TCL script for xcelium references:

```bash
cat /tmp/finn_dev_$(whoami)/code_gen_ipgen_StreamingMaxPool_hls_*/hls_syn_*.tcl
```

Potential environment variable workarounds to try:

```python
os.environ["SIM_XCELIUM_LIB"] = ""
os.environ["XCELIUM_ROOT"] = ""
```

### `SIMD >= MW/1024` assertion error (step 11 — step_hw_codegen)

**Cause:** The fully-connected layer's input width (MW) is too large relative to the SIMD parallelism FINN selected.
**Fix:** The notebook uses a compact 3-block architecture (MW=1728) and generates a `folding_config.json` with `SIMD=4` in Defaults. If you change the architecture, recompute: `SIMD >= ceil(MW / 1024)` and ensure SIMD evenly divides MW.

### `Undefined for ScaledIntType` (step 11 — step_hw_codegen)

**Cause:** FINN's HLS backend cannot generate code for quantization above 8-bit. Brevitas exports >8-bit as `ScaledIntType`, which has no Vivado HLS equivalent (the `get_hls_datatype_str()` method raises an exception unconditionally for this type).
**Fix:** Only synthesize 4-bit and 8-bit models. The Float model is a reference only. See FINN FAQ: "the way we create the hardware isn't typically practical for more than 4 bits." The `max_multithreshold_bit_width` parameter defaults to 8.

### `FINN only supports unsigned and non-narrow Quant nodes for Relu`

**Cause:** Using signed activation quantizers (`Int8ActPerTensorFloat`) for ReLU layers.
**Fix:** Use `Uint8ActPerTensorFloat` (unsigned) for all `QuantReLU` layers. Only the input `QuantIdentity` uses signed `Int8ActPerTensorFloat` (because MFCC values can be negative).

### `StreamingMaxPool needs ImgDim % PoolDim == 0`

**Cause:** Spatial dimensions not exactly divisible by the pool kernel size at every stage.
**Fix:** The notebook pads inputs to the nearest multiple of 8 (for 3 × `MaxPool2d(2)`: 2³ = 8). With N_MFCC=20 → 24, MFCC_FRAMES=282 → 288.

### Wrong shell flow type (would fail at step 17)

**Cause:** Using `ShellFlowType.VITIS_ALVEO` for a Zynq board.
**Fix:** Use `ShellFlowType.VIVADO_ZYNQ`. The KV260 is a Zynq UltraScale+ board, not a PCIe Alveo card. FINN's `templates.py` confirms `KV260_SOM` maps to `zynq_us+`.

### numpy upgrade breaks Brevitas

**Cause:** `tensorflow` or `librosa` pulls in a newer numpy as a transitive dependency, causing C ABI incompatibility with the Brevitas build.
**Fix:** Use `tensorflow-cpu` in requirements. Never include numpy directly. If broken, restore with: `pip install numpy==<finn_pinned_version>`.

### Stale container after hard crash

**Cause:** If the host crashes or Docker is killed, the `--rm` container may not be cleaned up.
**Fix:** `docker rm finn_dev_<username>` then start a new container with `bash run-docker.sh bash`.

## FINN Build Outputs

After a successful synthesis, each model variant produces:

```
finn_build/
  qat_4bit/
    report/          ← Resource estimates (LUT, BRAM, DSP, FF) and throughput
    stitched_ip/     ← Vivado IP core (AXI-Stream in/out)
    bitfile/         ← Bitstream .bit + hardware handoff .hwh
    deploy/          ← Deployment package with PYNQ driver
  qat_8bit/
    ...
```

## Deploying on the KV260

1. Copy the `deploy/` folder to the KV260 via `scp` or `rsync`
2. On the KV260, source the PYNQ environment
3. Use the generated Python driver to load the accelerator and run inference
4. Feed MFCC features (shape `1×2×24×288`) to the accelerator input
5. Read the classification output (speaker class index)
